# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

Public static site (GitHub Pages, served at `poseidonstats.com`) plus the Python clip-generation toolkit for TikTok/Instagram Reels. The site itself is intentionally minimal: 3 HTML pages + vanilla JS + 3 JSON files. All heavy lifting (Poisson model, ratings, simulations, backtest) lives **outside this repo**, in `~/football_predictor/` — a separate private codebase that publishes JSON here.

The repo is **public**. `.gitignore` is the safety net — anything that could leak the model (CSVs, scripts in `scripts/poseidon/`, predict_daily/ratings/bayesian_mc/dixon_coles, *.pkl/joblib, .env) is blocked. Before adding files to `data/` or `scripts/`, check `.gitignore` won't surprise you.

## Architecture

```
~/football_predictor/         (private pipeline — separate repo)
  scripts/poseidon/predict_daily.py  → poisson_advanced_predictions.csv
  scripts/poseidon/freeze_history.py → poseidon_history.csv (forward jurnal)
  scripts/poseidon/resolve_history.py → marks RESOLVED / FT scores
  scripts/poseidon/build_public_json.py → writes 3 JSONs into this repo's data/

~/poseidon-site/              (THIS repo — public)
  data/predictions.json       — generated by build_public_json.py
  data/calibration.json       — bucket calibration from backtest 65k
  data/forward.json           — live tracking from poseidon_history.csv
  assets/app.js               — renders all 3 pages, reads JSONs
  scripts/daily_publish.sh    — regen JSON + git push (cron 07:50)
  scripts/gen_*_clip.py       — TikTok/IG clip generators (PIL + ffmpeg)
  scripts/_v2_common.py       — shared helpers for clips
```

Pipeline runs daily on user's MacBook via LaunchAgents in `~/Library/LaunchAgents/com.poseidonstats.*.plist` and `com.football.poseidon.*.plist`. Site is rebuilt and pushed once per day by `daily_publish.sh` at 07:50 EEST.

### Picks logic — must stay in sync with `assets/app.js`

The "★ pick" suggestions on the site come from `pickBadge()` + the picks accumulator in `assets/app.js`. Thresholds:
- Over 1.5: ≥ 0.75 (note: stricter than the 0.65 baseline)
- Over 2.5, Over 3.5, Home, Away, BTTS Da, BTTS Nu, Under 2.5, Under 1.5, Under 3.5: ≥ 0.65
- Badge colour: `pick-elite` ≥ 0.80, `pick-strong` ≥ 0.70, `pick-good` ≥ 0.65

If you change these, **also update `gen_today_clip.py`'s `picks_of()`** — it replicates the site logic. The two drifted once (int vs round on percentages); always use `round(p*100)` to match.

There is no build step (vanilla JS, no bundler) and no test suite. Site changes are verified by `curl` + visual check on the live URL after `git push`. GitHub Pages propagation is 1–2 min; clear Service Worker cache if needed.

## Generated clips — non-obvious constraints

These rules came from real bugs that shipped to production. Apply them every time:

1. **Verify frame-by-frame before recommending a post.** Clips are auto-generated from CSV data — picks can include obscure leagues / women's teams / U17 teams / 99% inflated probabilities. Read at least the score, top-WIN, top-LOSS frames as PNGs before saying "ready to post".

2. **Display filters in `gen_yesterday_clip.py`** (`displayable()`):
   - `EXCLUDE_TEAM_SUFFIX` covers `II/III/B/C/Reserve(s)`, `U14–U23`, `W/Women/Ladies/Femenino/Femenil/Feminin`.
   - `EXCLUDE_LEAGUE_KEY` covers `women/femenil/femenino/feminin/ladies/w-league`.
   - `MAX_PROB_DISPLAY = 0.88` excludes inflated super-favourites from being shown as examples.
   The aggregate WIN/LOSS counters use the full sample; only the *displayed* example picks are filtered.

3. **`gen_today_clip.py` and `gen_wc_clip.py` use stricter allow-lists** (top-2 leagues + recognised friendlies for "today", calibrated Over 1.5 70–90% / HT Over 0.5 60–80% only for WC). These are the markets where calibration vs historical reality stays within ±5pp on national-team matches.

4. **Clips are MUTE.** Audio is added in the TikTok/IG app at upload (trending sounds). Do not embed audio.

5. **Loop perfect, hook < 3s, question at end.** First and last frame are the same `cadru_loop()` (trident + POSEIDON) so the auto-replay feels seamless. The penultimate frame is `cadru_intrebare()` to provoke comments (the TikTok algorithm weights comments heavily). Keep clips 15–18s.

## Honesty conventions (non-negotiable)

These are brand-defining and have failed quietly when ignored:

- **Show calibrated probability, not the raw model output.** When the brut model says 80% BTTS but the empirical bucket shows 67%, display 67% with the raw number as a footnote (`prob_btts_raw` in predictions.json). Site does this; clips must too.
- **Tier language matches sample size.** N < 30 = `noise`, N ≥ 100 = `promising`, N ≥ 300 = `robust`, N ≥ 500 + Wlo ≥ 70% = `strong robust`. Never say "robust" on N < 300. Use Wilson lower bound at 95%, not raw hit-rate.
- **Show LOSSes, not just WINs.** The verification clip shows the highest-probability LOSS as one of its frames. Removing this breaks the entire brand premise.
- **Zero gambling-site links.** Anywhere — site, clips, captions. Grep `href` before push: nothing matching `bet|cazino|casino|pariu|superbet|unibet|fortuna|bet365` should appear.
- **Disclaimer is mandatory on every page and every clip frame**: `informativ · 18+ · nu garanție`. `legal-banner` at the top of every HTML page. `add_disclaimer()` on every PIL frame.
- **Never embed a betting slip image or claim wins in EUR/RON.** This violates Meta/TikTok community guidelines and Romanian ONJN regulations.

## Site security setup (already applied)

- HTTPS via Let's Encrypt (auto-renewed by GitHub Pages)
- HSTS 1 year
- CSP meta tag in all 3 HTML pages (self + `gc.zgo.at` for GoatCounter only)
- Referrer-Policy: `strict-origin-when-cross-origin`
- CAA DNS records at datahost (issue letsencrypt.org only, issuewild blocked, iodef → contact email)
- GoatCounter for analytics (privacy-first, no cookies, no GDPR banner)

If you change the CSP, test it against the live site immediately — a too-strict CSP will silently break the JS. The current policy allows inline scripts (`'unsafe-inline'`) because index/track-record register the Service Worker and call `renderTrackRecord()` inline.

## Methodology page — what to keep and what to hide

`metodologie.html` is intentionally calibrated for transparency without giving away the exact parameters that would let someone replicate the model:

- **Keep**: Poisson formulas (`λ_home = attack × defense × home_adv × μ_league_calibrated`), Dixon-Coles concept, shrinkage Bayesian concept, Monte Carlo 200k+ iterations, league counts (220 calibrated / 94 blacklisted), bias examples (Nordic +22%, South America −22%), known limitations.
- **Remove if anyone adds them**: half-life value in days, shrinkage K value, prior value, exact rolling window for ρ (e.g. "5000 matches"), the literal formula `cal_scale_lg = real_goals / predicted_goals`.

Concept = credibility. Fine numbers = free uplift for a copier.

## When the site won't update after push

Likely cause: Service Worker is still serving v(N-1) from cache. Fix:
1. Bump `CACHE = "poseidon-vN"` in `sw.js`.
2. Commit + push.
3. On a phone: close the tab fully, wait 10s, reopen. SW will detect the new version and invalidate.

The publish script does not bump the cache automatically — you do it manually when an HTML/CSS/JS change needs to be visible immediately.

## Predictions JSON format

`predictions.json` matches have these keys (see `build_public_json.py` for source of truth):
```
fixture_id, match_date (ISO UTC), country, league, home_team, away_team,
xg_home, xg_away,
prob_home, prob_draw, prob_away,
prob_over_1_5, prob_over_2_5, prob_over_3_5,
prob_btts, prob_ht_over_0_5, prob_ht_over_1_5, prob_ht_btts,
ht_prob_home, ht_prob_draw, ht_prob_away,
ht_top_score_h, ht_top_score_a, ht_top_score_prob,
xg_home_ht, xg_away_ht,
prob_over_2_5_raw, prob_btts_raw,        # uncalibrated, for transparency
top_score_h, top_score_a, top_score_prob,
calibrated                                # bool — was empirically calibrated for this league
```

The HT market lives under `ht_prob_over_0_5`, **not** `prob_ht_over_0_5` — these names are easy to confuse.
